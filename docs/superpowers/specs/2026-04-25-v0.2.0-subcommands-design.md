# v0.2.0 — S3 API subcommand expansion

Status: approved (brainstorming session, 2026-04-25)
Branch: `v0.2.0`

## Overview

Add 13 new subcommands to `s3util`, all of which are thin wrappers over single AWS S3 API calls. The existing `cp` and `mv` subcommands (multipart transfer pipeline) are unaffected.

The new commands fall into four functional groups:

- **Object operations** (5): `head-object`, `rm`, `get-object-tagging`, `put-object-tagging`, `delete-object-tagging`
- **Bucket lifecycle** (3): `create-bucket`, `head-bucket`, `delete-bucket`
- **Bucket versioning** (2): `put-bucket-versioning`, `get-bucket-versioning`
- **Bucket policy** (3): `put-bucket-policy`, `get-bucket-policy`, `delete-bucket-policy`

`head-bucket` is implemented first as the foundation PR — it exercises the whole new framework (CLI args, common args, storage wrapper, JSON output) on the simplest possible API call.

## Goals

- One thin S3-API command per subcommand. No transfer pipeline, no progress indicator, no multipart logic.
- JSON output for read commands (`head-*`, `get-*`) in **AWS CLI v2 `--output json` shape**: PascalCase keys, RFC3339 timestamps, integer/string types matching botocore, optional fields *omitted* when absent (never `null`).
- Silent on success for write commands (exit 0, nothing on stdout).
- Each command exposes the common option groups listed below as subcommand-level (not global) flags.
- Existing `cp` and `mv` are not refactored. Their behaviour, exit codes, flags, and tests stay byte-identical.

## Non-goals

- No recursive / bulk operations (`s3util` remains single-target by design — see README "Scope").
- No MFA-delete on `put-bucket-versioning` (niche; can be added later).
- No conditional headers / `Range` / `PartNumber` on `head-object` (rarely used; can be added later).
- No automatic rollback if `create-bucket --tagging` succeeds at create-time but fails at tag-time. The bucket is left created-and-untagged with a stderr warning and exit 3.
- No SIGINT cleanup for the new commands (single API call; no in-flight state to abort).

## Common option groups

Every new subcommand flattens a new `CommonClientArgs` struct (in `src/config/args/common_client.rs`) containing exactly:

| Group | Flags |
|---|---|
| Tracing/Logging | `--json-tracing`, `--aws-sdk-tracing`, `--span-events-tracing`, `--disable-color-tracing` |
| AWS Configuration | `--aws-config-file`, `--aws-shared-credentials-file`, `--target-profile`, `--target-access-key`, `--target-secret-access-key`, `--target-session-token`, `--target-region`, `--target-endpoint-url`, `--target-force-path-style`, `--target-accelerate`, `--target-request-payer`, `--target-no-sign-request`, `--disable-stalled-stream-protection` |
| Retry Options | `--aws-max-attempts`, `--initial-backoff-milliseconds` |
| Timeout Options | `--operation-timeout-milliseconds`, `--operation-attempt-timeout-milliseconds`, `--connect-timeout-milliseconds`, `--read-timeout-milliseconds` |
| Advanced | `--auto-complete-shell` |

Verbosity (`-v`, `-q`) is included in `CommonClientArgs` for parity with cp/mv.

For commands operating on a single S3 endpoint, the auth/region/endpoint flags use the `--target-*` prefix (the convention chosen for consistency across the new commands). Version targeting on object operations uses `--source-version-id` (mirrors cp/mv's source-side naming for the version of the object being acted on).

`CommonTransferArgs` (cp/mv) is **not refactored**. Some flag definitions are duplicated between the two structs; deduplication may happen in a later release.

All commands flatten `CommonClientArgs`. The table below lists positional arguments and command-specific flags only.

| # | Command | Positional | Specific flags | Output |
|---|---|---|---|---|
| 1 | `head-object` | `s3://bucket/key` | `--source-version-id <ID>`, `--enable-additional-checksum`, `--source-sse-c <ALG>`, `--source-sse-c-key <KEY>`, `--source-sse-c-key-md5 <MD5>` | JSON |
| 2 | `rm` | `s3://bucket/key` | `--source-version-id <ID>` | silent |

| 4 | `head-bucket` | `s3://bucket` | — | JSON |
| 5 | `delete-bucket` | `s3://bucket` | — | silent |
| 6 | `put-bucket-versioning` | `s3://bucket` | `--enabled` \| `--suspended` (clap `ArgGroup`, exactly one required) | silent |
| 7 | `get-bucket-versioning` | `s3://bucket` | — | JSON |
| 8 | `put-bucket-policy` | `s3://bucket POLICY_FILE\|-` | — | silent |
| 9 | `get-bucket-policy` | `s3://bucket` | — | JSON |
| 10 | `delete-bucket-policy` | `s3://bucket` | — | silent |
| 11 | `get-object-tagging` | `s3://bucket/key` | `--source-version-id <ID>` | JSON |
| 12 | `put-object-tagging` | `s3://bucket/key` | `--tagging "k=v&k2=v2"` (required), `--source-version-id <ID>` | silent |
| 13 | `delete-object-tagging` | `s3://bucket/key` | `--source-version-id <ID>` | silent |

### Per-command behavioural notes

- **`head-object --enable-additional-checksum`** — sets `ChecksumMode=ENABLED` on the request so S3 includes the additional checksum in the response, which appears in the JSON output (e.g. `ChecksumSHA256`, `ChecksumCRC32`). Without the flag, those fields are omitted from the JSON.
- **`head-object` SSE handling** — only `--source-sse-c-*` is exposed. SSE-S3 (`AES256`) and SSE-KMS objects need no client-side flags; their SSE state appears in the JSON response as `ServerSideEncryption`, `SSEKMSKeyId`, `BucketKeyEnabled`. The `--source-sse-c-*` triple is required when the object was written with SSE-C, otherwise S3 returns 400.
- **`rm`** — single object only. With `--source-version-id`, deletes a specific version. Without, creates a delete marker (on a versioned bucket) or removes the object (on a non-versioned bucket).
- **`create-bucket`** — `LocationConstraint` is derived from the SDK client's resolved region (which honours `--target-region`, `AWS_REGION`, and the active profile's region in that order); deriving from `--target-region` alone caused `IllegalLocationConstraintException` whenever the region came from env or profile. The `us-east-1` quirk (no `LocationConstraint` element allowed) is handled inline. With `--tagging`, the implementation makes two API calls: `CreateBucket` then `PutBucketTagging`. If `PutBucketTagging` fails after `CreateBucket` succeeds, the bucket exists untagged; we exit **3** with a stderr warning explaining the partial state. **No automatic rollback** — an automatic delete-on-failure could itself fail and produce worse outcomes.
- **`head-bucket`** — JSON output contains the headers S3 returns: `BucketRegion`, `AccessPointAlias` (and any others surfaced by `aws-sdk-s3`). Mirrors `aws s3api head-bucket --output json`.
- **`put-bucket-versioning`** — clap `ArgGroup` enforces `--enabled XOR --suspended`. Maps to `VersioningConfiguration.Status = Enabled | Suspended`.
- **`get-bucket-versioning`** — emits `{}` when the bucket has never had versioning configured (S3 returns no `Status` element). Otherwise emits `{"Status": "Enabled" | "Suspended", "MFADelete": "..." }` as fields are present.
- **`put-bucket-policy`** — second positional argument is a path or `-`. With a path, reads the file verbatim. With `-`, reads from stdin until EOF. The body is sent as the policy JSON without parsing or validation by `s3util` (S3 will reject invalid policies with 400).
- **`get-bucket-policy`** — output mirrors `aws s3api get-bucket-policy` exactly: `{"Policy": "<escaped-JSON-string>"}`. The double-encoding is intentional — pipes through `jq -r .Policy | jq` to pretty-print.
- **`get-object-tagging`** — emits `{"TagSet": [{"Key": "...", "Value": "..."}], "VersionId": "..."}`. Empty `TagSet` is `[]`, not omitted.
- **`put-object-tagging`** — `--tagging` is required. Replaces all tags on the object (S3 semantics).
- **`delete-object-tagging`** — removes all tags. With `--source-version-id`, targets a specific version.

## Output module (`src/output/json.rs`)

Hand-written `serde_json::Value` builders, one per JSON-emitting command:

```rust
pub fn head_object_to_json(out: &HeadObjectOutput) -> Value
pub fn head_bucket_to_json(out: &HeadBucketOutput) -> Value
pub fn get_bucket_versioning_to_json(out: &GetBucketVersioningOutput) -> Value
pub fn get_object_tagging_to_json(out: &GetObjectTaggingOutput) -> Value
pub fn get_bucket_policy_to_json(out: &GetBucketPolicyOutput) -> Value
```

Conventions:

- PascalCase keys (matching the AWS CLI surface, not the SDK's snake_case fields).
- Timestamps as RFC3339 strings, e.g. `"2024-01-15T08:30:00+00:00"`.
- Sizes/counts as JSON numbers (integers, unquoted).
- Optional fields omitted when absent.
- Output is pretty-printed via `serde_json::to_string_pretty`, terminated with a newline, written to stdout.

`aws-sdk-s3` response types do not implement `Serialize`, and their internal field shapes don't match the AWS CLI surface 1-to-1 (e.g. PascalCase, omission semantics, double-encoded `Policy`). Hand-written mappers — ~30 lines each — give exact control over the documented shape.

## Storage layer (`src/storage/s3/api.rs`)

14 thin async wrappers, each ~15-25 lines:

```rust
pub async fn head_object(client: &Client, bucket: &str, key: &str, opts: HeadObjectOpts) -> Result<HeadObjectOutput>
pub async fn delete_object(client: &Client, bucket: &str, key: &str, version_id: Option<&str>) -> Result<DeleteObjectOutput>
pub async fn create_bucket(client: &Client, bucket: &str, region: &str) -> Result<CreateBucketOutput>
pub async fn put_bucket_tagging(client: &Client, bucket: &str, tagging: Tagging) -> Result<PutBucketTaggingOutput>
pub async fn head_bucket(client: &Client, bucket: &str) -> Result<HeadBucketOutput>
pub async fn delete_bucket(client: &Client, bucket: &str) -> Result<DeleteBucketOutput>
pub async fn put_bucket_versioning(client: &Client, bucket: &str, status: VersioningStatus) -> Result<PutBucketVersioningOutput>
pub async fn get_bucket_versioning(client: &Client, bucket: &str) -> Result<GetBucketVersioningOutput>
pub async fn put_bucket_policy(client: &Client, bucket: &str, policy: &str) -> Result<PutBucketPolicyOutput>
pub async fn get_bucket_policy(client: &Client, bucket: &str) -> Result<GetBucketPolicyOutput>
pub async fn delete_bucket_policy(client: &Client, bucket: &str) -> Result<DeleteBucketPolicyOutput>
pub async fn get_object_tagging(client: &Client, bucket: &str, key: &str, version_id: Option<&str>) -> Result<GetObjectTaggingOutput>
pub async fn put_object_tagging(client: &Client, bucket: &str, key: &str, version_id: Option<&str>, tagging: Tagging) -> Result<PutObjectTaggingOutput>
pub async fn delete_object_tagging(client: &Client, bucket: &str, key: &str, version_id: Option<&str>) -> Result<DeleteObjectTaggingOutput>
```

Each wrapper builds the SDK operation, calls `.send().await`, maps the SDK error to `anyhow` with operation context. The `Client` is built via the existing `storage/s3/client_builder.rs` (parameterised to accept either `CommonTransferArgs` or `CommonClientArgs`).

## Error handling and exit codes

| Code | When |
|---|---|
| 0 | Success |
| 1 | S3 error (AccessDenied, BucketNotEmpty, NoSuchKey/NoSuchBucket on commands not listed under exit 4, region mismatch on `head-*`, etc.), IO error, validation error |
| 2 | clap argument-parse error |
| 3 | `create-bucket --tagging` succeeded the create step but failed the follow-up `PutBucketTagging`; bucket exists untagged |
| 4 | NotFound on `head-bucket`, `head-object`, `get-object-tagging`, `get-bucket-policy`, `get-bucket-tagging`. Specifically: `head-*` use the SDK's `is_not_found()` (404 NoSuchBucket / NoSuchKey / NoSuchVersion); the three `get-*` match the S3 error code — `get-object-tagging`: `NoSuchKey` / `NoSuchBucket` / `NoSuchVersion`; `get-bucket-policy`: `NoSuchBucket` / `NoSuchBucketPolicy`; `get-bucket-tagging`: `NoSuchBucket` / `NoSuchTagSet`. Other failure modes on these commands (auth, region mismatch, SSE-C key error) still exit 1. |
| 130 | Reserved for cp/mv (not produced by the new commands) |
| 101 | Internal panic |

All SDK errors except the NotFound cases listed above are wrapped with `anyhow::Context` describing the operation (e.g. `"head-object on s3://bucket/key"`). The S3 error code/message is preserved in the chain so users can grep stderr for `NoSuchKey`, `AccessDenied`, etc. The NotFound exit code is special-cased per S3 error class — added so scripts can distinguish "doesn't exist" (including "no policy attached" for `get-bucket-policy` and "no tags configured" for `get-bucket-tagging`) from generic operation failures without parsing stderr.

## Testing

- **Unit tests** (no AWS) in `src/config/args/tests.rs` (extending existing patterns): args parsing, defaulting, mutually-exclusive groups (`--enabled`/`--suspended`), required-args validation.
- **Unit tests for `output/json.rs`**: feed synthetic SDK output, assert exact JSON shape — key names, omitted-vs-present fields, RFC3339 timestamp format. Catches regressions in the AWS-CLI-shape mapping without needing AWS.
- **CLI tests** (no AWS) following `tests/cli_arg_validation.rs` / `tests/cli_config_validation_error.rs`: e.g. `put-bucket-versioning` with neither `--enabled` nor `--suspended` → exit 2; `put-bucket-policy` with non-existent file → exit 1.
- **E2E tests** (gated behind `RUSTFLAGS=--cfg e2e_test`, **user-run only**), one file per command-group:
  - `tests/e2e_head_object.rs`
  - `tests/e2e_rm.rs`
  - `tests/e2e_create_delete_bucket.rs`
  - `tests/e2e_head_bucket.rs`
  - `tests/e2e_bucket_versioning.rs`
  - `tests/e2e_bucket_policy.rs`
  - `tests/e2e_object_tagging.rs`

E2E tests will not be executed during development; instead, every PR verifies the e2e modules compile cleanly with `RUSTFLAGS="--cfg e2e_test" cargo check --all-features` (and `clippy` under the same flag).

## Implementation phasing

Five PRs, each self-contained (compiles, unit/CLI tests pass, e2e modules compile under `--cfg e2e_test`).

**PR 1 — Foundation: `head-bucket` end-to-end**
Establishes `CommonClientArgs`, `output/`, `storage/s3/api.rs`, and ships `head-bucket` as proof of the framework.

**PR 2 — Object operations** (5 commands)
`head-object`, `rm`, `get-object-tagging`, `put-object-tagging`, `delete-object-tagging`.

**PR 3 — Bucket lifecycle** (2 commands)
`create-bucket` (with `--tagging` two-step), `delete-bucket`.

**PR 4 — Bucket versioning** (2 commands)
`put-bucket-versioning`, `get-bucket-versioning`.

**PR 5 — Bucket policy** (3 commands)
`put-bucket-policy`, `get-bucket-policy`, `delete-bucket-policy`.

Each PR ends with `cargo fmt`, `cargo clippy --all-features` (zero warnings), and `RUSTFLAGS="--cfg e2e_test" cargo check --all-features` clean.

## Documentation

Each PR updates the README's command index and per-subcommand documentation. The README's existing structure (Features, Usage, Detailed information, Advanced options, All command-line options) accommodates the new commands without restructuring.

## Out of scope for v0.2.0

Tracked here so a future release can pick them up without re-litigating:

- MFA-delete on `put-bucket-versioning`.
- Conditional headers (`If-Match`, `If-None-Match`, `If-Modified-Since`, `If-Unmodified-Since`), `Range`, `PartNumber` on `head-object`.
- Object-lock options on `create-bucket`.
- Canned ACL flag on `create-bucket` (only tagging is in scope).
- Deduplication of overlapping fields between `CommonTransferArgs` and `CommonClientArgs`.
- A bulk/multi-object form of `rm` (would conflict with the single-object scope statement in the README).
